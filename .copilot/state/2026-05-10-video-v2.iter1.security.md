> Reviewer: Security · Date: 2026-05-10

## Scope of this review

This is a plan-time review per `.copilot/agents/planner.md` ("flag anything touching auth, secrets, PII, or dependencies — the Security agent must review the plan before build starts"). V2 introduces:

- the **first untrusted byte stream** into the API (multipart upload),
- a **server-side HMAC secret** (`Features:Video:SigningSecret`),
- **filesystem writes** under a configurable root,
- a **public, redirect-issuing endpoint** (`/stream`) that mints credentials,
- **rate limiting** and **per-team quota** as the only line of defence against DoS and disk-fill.

Therefore it is in scope on at least 7 of the 10 non-negotiables.

## Findings

### F1 — `Referrer-Policy: no-referrer` on stream redirects (severity: low)

The 302 mint endpoint hands the browser a signed URL with the secret-derived `sig` query param. Without an explicit policy, a `<video src=…>` followed by a `fetch()` from the page can leak the URL via `Referer`.

**Remediation (must be in V2 backend):** the 302 response from `/stream` (mint) and the 200 streaming response from `/stream?…&sig=…` MUST set:

```
Referrer-Policy: no-referrer
Cache-Control: private, no-store
```

Add an integration test that asserts both headers are present.

### F2 — Multipart upload request timeout (severity: low → medium)

`MultipartReader` with `Request.BodyReader` will happily wait forever for the next byte; a slow-loris client can hold a Kestrel connection open with the rate-limit budget already spent on the current request.

**Remediation:** add `Kestrel.Limits.RequestHeadersTimeout` (default already 30 s, document it) and an explicit `IHttpRequestTimeoutFeature` of 5 minutes scoped to the upload endpoint via `RequestTimeoutsMiddleware`. 5 minutes is enough for 500 MiB at 14 Mbit/s (3G), tight enough that a stalled connection drops.

Test: a faked `Stream` that yields one byte every 30 s causes the request to be cancelled before quota math runs.

### F3 — Free-space check pre-write, not just at startup (severity: low)

The plan checks free disk at startup. That's necessary but not sufficient — disk usage drifts. Before `File.Move(tmp, final)` the store must check `DriveInfo.GetDrives().AvailableFreeSpace > MinFreeBytes` (default 1 GiB) and return a deterministic 503 (`storage_unavailable`) if not.

**Remediation:** add a check inside `LocalFsObjectStore.PutAsync` before the `File.Move`. Test with a stub `IFreeSpaceProbe`.

### F4 — Reject the example signing secret in non-test environments (severity: low)

`appsettings.json` ships `SigningSecret = ""`. Good. But once a developer copies a secret like `"change-me-please"` into `appsettings.Development.json` and forgets to rotate it, that value can ride into prod via templating mistakes.

**Remediation:** at startup, in `IHostEnvironment.IsProduction()`, refuse to start if the secret matches any string in a tiny deny-list (`""`, `"change-me"`, `"change-me-please"`, `"dev-secret"`, the literal `"forgerise-dev"`). Test asserts startup fails when prod env + denied secret.

### F5 — `RawStorageKey` must be a relative key, not joinable to `..` (severity: medium → low because mitigated)

The plan already says "storage key is a relative key … never an absolute path" (Approach §4). To make this a hard guarantee in code:

**Remediation:** `LocalFsObjectStore.Resolve(string key)` must:
1. Reject keys containing `\\`, `..`, `:`, or starting with `/`.
2. Compute `var full = Path.GetFullPath(Path.Combine(Root, key));`
3. Assert `full.StartsWith(Path.GetFullPath(Root) + Path.DirectorySeparatorChar)` — otherwise throw.

Test the matrix: `"../../../etc/passwd"`, `"teams/x/../../../y"`, `"C:\\Windows\\system32\\sam"`, `"\0"`, empty.

### F6 — `IUploadService` must hash the stored bytes for later AV / dedup (severity: info)

V2 explicitly defers AV scanning to V3. To keep that a one-line change later, `UploadService` should compute a SHA-256 over the upload stream as it writes (single pass, free) and store it on `VideoAsset.ContentSha256` (new column).

**Remediation:** plan addendum — V2 adds a `ContentSha256` column to `VideoAsset` via a small migration (`AddVideoAssetContentSha256`). V3's AV scanner reads this hash to dedup work; today it gives us a second line of forensic defence.

This is the only schema change V2 introduces. It is small, additive, and reversible.

## Standing rules carried forward (from V1 review, still apply)

1. `TeamScope.RequireOwnedTeam` on every endpoint that takes a `teamId`.
2. Signed URL HMAC must bind viewer; TTL ≤ 15 min.
3. ffprobe MIME validation BEFORE any byte is treated as "ready" — V2 substitutes a 12-byte ftyp sniff as a partial mitigation, with the gap explicitly accepted in Risks §6.
4. Worker uses a separate DB role (deferred to V3 since no worker yet — re-flag at V3 plan time).
5. AI prompts via `IInsightService` excluding welfare data (V5+).
6. No PII in logs.
7. Any new dependency requires a fresh review.

## Dependency check

Plan introduces zero new NuGet packages. `Microsoft.AspNetCore.RateLimiting` is in the framework. `Microsoft.AspNetCore.Http.Timeouts` (for F2) is in the framework. **Pass.**

## Decision

**PASS with conditions.** The 6 findings (F1-F6) are absorbed into the backend implementation. None individually is a blocker; collectively they are the cost of crossing the trust boundary. Backend may proceed to tasks 2-6 once this file is committed.

Re-review trigger: if backend deviates from the plan on any of (HMAC construction order, TTL cap, MIME sniff list, quota transaction shape, secret deny-list), Security must be re-invoked before merge.
