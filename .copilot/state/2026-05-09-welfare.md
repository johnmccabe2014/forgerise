## Goal

Backend welfare-and-readiness workflow that lets a coach record well-being signals against a player, derives a coach-safe readiness category, exposes only the safe view to the team feed, audits every read of raw fields, and supports GDPR-friendly retention.

## Constraints (master prompt §9)

- Raw welfare fields stay server-side; coach UI sees only `SafeCategory` (`Ready`/`Monitor`/`ModifyLoad`/`RecoveryFocus`).
- No raw welfare in logs (handled already by `WelfareDestructuringPolicy` from Phase 1).
- No medical diagnosis. Categorisation is a heuristic, not a clinical assessment.
- Every read of raw fields is written to a `WelfareAuditLog` row.
- Soft-delete + retention policy on raw notes (purgeable separately from category history).

## Deliverables

### A. Data
- `WellnessCheckIn` entity (PlayerId, AsOf, raw scores, derived `SafeCategory` snapshot, `DeletedAt?`, `RawPurgedAt?`).
- `IncidentReport` entity (PlayerId, OccurredAt, Severity, Summary [coach-safe], Notes [raw], `DeletedAt?`, `RawPurgedAt?`).
- `WelfareAuditLog` (Id, ActorUserId, PlayerId, Action enum, At).
- DbSets + indexes + soft-delete query filters; EF migration `Welfare`.

### B. Domain logic
- `ReadinessCategorizer` pure function: `(sleepHours, sorenessScore, moodScore, stressScore, fatigueScore) → SafeCategory`. Documented thresholds. Unit-tested with table cases.

### C. Endpoints (all `[Authorize]`, all use existing OwnsTeam pattern)
- `POST /teams/{teamId}/players/{playerId}/checkins` — record raw fields; categorise; persist; emit `welfare.checkin.recorded` (no raw).
- `GET /teams/{teamId}/readiness` — list `(playerId, displayName, category, asOf)` from latest non-deleted checkin per player. **No raw**.
- `GET /teams/{teamId}/players/{playerId}/checkins/{id}/raw` — owner-only; writes a `WelfareAuditLog` row of `Action=ReadRaw` before returning.
- `DELETE /teams/{teamId}/players/{playerId}/checkins/{id}` — soft-delete (retains category snapshot); `welfare.checkin.deleted`.
- `POST /teams/{teamId}/players/{playerId}/checkins/{id}/purge-raw` — explicit GDPR purge (clears raw scores + notes, sets `RawPurgedAt`); audited.
- `POST /teams/{teamId}/players/{playerId}/incidents` — record incident.
- `GET /teams/{teamId}/incidents` — list (Summary only, never Notes).
- `GET /teams/{teamId}/incidents/{id}/raw` — owner-only, audited.
- `GET /teams/{teamId}/audit` — owner only; lists welfare audit rows for the team's players.

### D. Tests (xUnit)
- `ReadinessCategorizerTests` — table-driven (≥6 cases including boundary).
- `WelfareEndpointsTests` — owner CRUD, non-owner 403, readiness shows safe category only, raw-read writes audit row, purge-raw clears fields.
- `IncidentEndpointsTests` — list omits notes, raw read is audited.
- `WelfareLoggingTests` — using a Serilog test sink, asserts a structured log with raw fields gets the values redacted by the destructuring policy.
