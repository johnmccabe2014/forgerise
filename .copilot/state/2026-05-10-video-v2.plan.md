# Plan — Video V2: real upload + LocalFsObjectStore + signed-URL playback

> Owner: Planner · Task ID: video-v2 · Date: 2026-05-10
> Predecessor: video-v1 (commit 67211b6, APPROVED 37/40)
> Architecture ref: [docs/video-intelligence-architecture.md](../../docs/video-intelligence-architecture.md)

## 1. Problem statement

V1 shipped a flag-gated, empty Video module. V2 introduces the **first untrusted byte stream** into the API: a coach uploads an `.mp4`, the file lands on disk under a per-team prefix, and the coach can play it back via a short-lived signed URL. No transcoding, no ffprobe, no AI yet — those slices land in V3+. The goal of V2 is to prove the **storage + auth + rate-limit boundary** end-to-end with a boring local backend, so the abstraction (`IObjectStore`) and the auth contract (signed URLs HMAC-bound to `viewerUserId` + `expiresAt`) are road-tested before we add S3/Azure Blob and ffmpeg.

## 2. Acceptance criteria

- [ ] **AC1** — A team member can `POST /v1/teams/{teamId}/videos/uploads` with a multipart body containing a single `.mp4` ≤ `Features:Video:MaxUploadBytes` (default 500 MiB) and receive `201 { videoAssetId, processingState: "Queued" }`. The bytes are persisted under `{Storage:Root}/teams/{teamId}/raw/{videoAssetId}.mp4`.
- [ ] **AC2** — `LocalFsObjectStore` implements `IObjectStore` and is the sole DI registration of `IObjectStore` for V2. Writes use `FileMode.CreateNew` (no overwrite), reads stream with bounded buffers, deletes are best-effort idempotent.
- [ ] **AC3** — `GetSignedReadUrlAsync(storagePath, viewerUserId, expiresAt, ct)` returns a URL of shape `/v1/teams/{teamId}/videos/{videoId}/stream?exp={unix}&v={viewerUserId}&sig={hex}` where `sig = HMAC-SHA256(secret, "{storagePath}|{viewerUserId}|{exp}")`. TTL > 15 min throws `ArgumentOutOfRangeException`.
- [ ] **AC4** — `GET /v1/teams/{teamId}/videos/{videoId}/stream` (no `sig` param): 302 to a freshly minted signed URL whose `v` matches the calling user's id; only team members can request this.
- [ ] **AC5** — `GET …/stream?exp=…&v=…&sig=…`: 200 + `video/mp4` body when the HMAC verifies AND `exp` is in the future AND `v` matches a current member of the team that owns the asset. 401 (`signature_invalid`) if HMAC fails. 410 (`signature_expired`) if `exp` < now. 403 if `v` is no longer a team member.
- [ ] **AC6** — MIME allow-list is enforced **at upload time before the bytes are written to disk**: sniff the first 12 bytes for an ISO-BMFF `ftyp` box (`mp4`/`isom`/`avc1`/`mp42`/`m4v `). Anything else → 415 with `{ code: "unsupported_media_type" }` and zero bytes touch the storage root.
- [ ] **AC7** — Per-team quota: total `VideoAsset.SizeBytes` (excluding soft-deleted) for a team must stay ≤ `Features:Video:TeamQuotaBytes` (default 5 GiB). Exceeding the quota → 413 with `{ code: "team_quota_exceeded", quotaBytes, usedBytes }`. Per-IP rate limit on the upload endpoint: 10 uploads / 5 min using the existing `Microsoft.AspNetCore.RateLimiting`.
- [ ] **AC8** — Listing endpoint from V1 now returns the uploaded asset (still `ProcessingState=Queued` because no worker yet); cross-team isolation tests from V1 still pass; flag-off behaviour from V1 still returns 404 on every new endpoint.

## 3. Non-goals

- **No** ffprobe / ffmpeg / transcoding — V3.
- **No** worker / background service / `IVideoJobQueue` impl — V3.
- **No** chunked or resumable upload — single-shot only; resumable is a V2.5 if grassroots actually needs it (most clips will be < 200 MiB, which is fine over 4G).
- **No** S3/Azure Blob backend — `LocalFsObjectStore` only; cloud backends slot behind `IObjectStore` later without touching the controller.
- **No** UI — the web upload page is V4.
- **No** voice notes, timeline events, transcripts, AI insights — later slices.
- **No** anti-virus scan — defer to V3 once the worker harness exists; documented as a known gap.

## 4. Approach

### Chosen

- **`LocalFsObjectStore`**: synchronous filesystem under a config-driven root. Two-step write — stream to `{root}/.uploads-tmp/{guid}` then `File.Move(tmp, final, overwrite:false)` — gives us atomic visibility and crash safety. Storage path is a relative key (`teams/{teamId}/raw/{videoAssetId}.mp4`), never an absolute path; the store is the only thing that joins it with `Root`.
- **Signed URL = HMAC over `{storagePath}|{viewerUserId}|{exp}`** with a server-side secret `Features:Video:SigningSecret` (32-byte minimum, validated at startup). HMAC is constant-time-compared. TTL hard-capped at 15 min in code, default 5 min.
- **Upload happens in the controller, not in a background job.** The 500 MiB cap is small enough to absorb in-process; the request streams from `HttpRequest.BodyReader` straight into the object store, never buffered fully in memory. Multipart parsing uses `MultipartReader` (no `[FromForm]` — that buffers).
- **Rate limit via existing ASP.NET Core rate limiter**, partitioned by `RemoteIpAddress`. Quota is a single `SUM(SizeBytes)` query inside the same transaction that creates the `VideoAsset` row, with a row-level `SELECT … FOR UPDATE` on a sentinel team row to serialise concurrent uploads from the same team (acceptable: grassroots teams upload one or two videos a week).
- **DTO shape stays the same as V1** (`VideoListItemDto`) so the listing endpoint just starts returning rows.

### Rejected alternatives

- **_Direct-to-S3 presigned uploads from the browser_** — pretty, but premature. We have no S3 yet, and adding it now means signing infra + CORS pain + a credentials story. `IObjectStore` lets us slot it in later.
- **_Chunked upload with `VideoUploadSession.BytesReceived`_** — the V1 entity supports it, but we save complexity by not exercising the chunk-handling path until a coach actually loses a 200 MiB upload at 90% on a wet pitch. Single-shot first; resumable is a measured-need follow-up.
- **_Use Postgres LOB / bytea storage_** — would simplify backups but ruins streaming semantics and balloons the DB. Filesystem is the boring choice; cloud blob behind the same interface comes for free in V5.
- **_Skip MIME sniffing, trust `Content-Type`_** — header is attacker-controlled. Sniff is 12 bytes of work and stops a class of bugs.

## 6. Risks & mitigations

| Risk                                                               | Likelihood | Impact   | Mitigation                                                                                                                                       |
|--------------------------------------------------------------------|-----------|----------|--------------------------------------------------------------------------------------------------------------------------------------------------|
| Signed-URL secret leaks via logs                                   | Low       | High     | Never log the secret or any signed URL. Test asserts `secret` substring not present in any log capture during a request.                        |
| Path traversal via crafted filenames                               | Low       | High     | Storage key is server-generated `{videoAssetId}.mp4`. Original filename is stored in DB only, never used as a path component.                    |
| MIME sniff bypass (mp4 inside polyglot)                            | Medium    | Medium   | Sniff is necessary-not-sufficient; V3 ffprobe is the authoritative gate before `ProcessingState=Ready`. Document this gap in the security file.  |
| Disk fills up                                                      | Medium    | High     | Per-team quota + global `Storage:Root` disk-usage check at startup; refuse uploads if free < 1 GiB. Logged via existing health check.            |
| Race between two concurrent uploads exceeding quota                | Low       | Low      | Quota check + insert in one transaction with row-lock on the team's `Team` row; second upload fails with 413.                                    |
| Streaming a 500 MiB file blocks Kestrel                            | Low       | Medium   | Use `Response.RegisterForDispose` + `FileStream` async copy with `useAsync:true`; ASP.NET pipes the stream — no full buffering.                   |
| Signed URL replay after user is removed from team                  | Medium    | Medium   | Stream verifier re-checks team membership of `v` on every request, not just at mint time. AC5 covers this.                                        |
| Test mp4 fixture trips antivirus on dev machines                   | Low       | Low      | Build the mp4 byte buffer in test code, not as a tracked binary. Total length < 100 bytes.                                                       |

5. Manually grep the diff for: any `IObjectStore` registration other than `LocalFsObjectStore`; any logging of `SigningSecret`; any path string concatenation outside `LocalFsObjectStore`; any `FileMode.Create` (must be `CreateNew`).
6. Confirm `appsettings.json` ships `Features:Video:Enabled = false` and a placeholder `SigningSecret = ""` (so prod can't accidentally start with a default secret).

## 8. Rollback plan

- **Code:** revert the V2 commit; V1 is untouched and continues to serve `404`s with the flag off.
- **Data:** the V1 schema is reused; no new migration in V2 (we're filling rows in tables that already exist). If a V2 upload partially succeeded and left an orphaned file, a one-shot SQL `DELETE FROM "VideoAssets" WHERE "CreatedAt" >= '<deploy-ts>'` plus `rm -rf {Storage:Root}/teams` cleans state. Flag stays `false` in prod until V2 is re-shipped.
- **Config:** `Features:Video:SigningSecret` is per-environment; rotate it by setting the env var and restarting — all outstanding signed URLs become 401, which is the correct behaviour after a key compromise.
