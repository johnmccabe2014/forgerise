# Phase 4 — Sessions, Attendance, AI Session Plan

**Date:** 2026-05-09
**Status:** in progress (iter1)
**Master prompt slice:** §8 first vertical slice — coach records attendance → writes session review → receives AI-generated next-session plan.

## Goals

Deliver the smallest end-to-end coach loop that turns attendance + a short session review (+ welfare readiness from Phase 3) into a deterministic, coach-friendly **next session plan**. AI is via an interface so OpenAI / Azure OpenAI / future providers can drop in later (master prompt §4); iter1 ships an in-process deterministic generator only.

Out of scope for this iteration: video uploads, match packs, push notifications, "highlight candidate" generation, calendar sync.

## Non-goals (carry forward)

- Real LLM call — only an in-process `HeuristicSessionPlanGenerator`. The interface is shaped so a `OpenAiSessionPlanGenerator` slots in later without controller changes.
- Multi-team scheduling / availability coordination across teams.
- Tactical pattern-of-play AI.

## A. Data

New entities (under `api/ForgeRise.Api/Data/Entities/`):

1. `Session`
   - `Id` (Guid), `TeamId` (FK), `ScheduledAt` (DateTimeOffset), `DurationMinutes` (int), `Type` (`SessionType {Training, Match, Other}`), `Location?` (string ≤120), `Focus?` (string ≤200, coach-facing one-liner), `ReviewNotes?` (string ≤4000, free-form coach review post-session), `ReviewedAt?` (DateTimeOffset), `CreatedByUserId`, `CreatedAt`, `DeletedAt?`.

2. `AttendanceRecord`
   - `Id`, `SessionId` (FK cascade), `PlayerId` (FK cascade), `Status` (`AttendanceStatus {Present, Absent, Late, Excused}`), `Note?` (string ≤500), `RecordedByUserId`, `RecordedAt`. Unique index on `(SessionId, PlayerId)`.

3. `SessionPlan`
   - `Id`, `TeamId`, `GeneratedAt`, `GeneratedByUserId`, `BasedOnSessionId?` (the most recent reviewed session that informed the plan), `Focus` (string ≤200), `Summary` (string ≤2000, coach-readable narrative), `PlanJson` (string, structured drills payload — array of `{block, title, durationMinutes, intent, intensity}`), `ReadinessSnapshotJson` (string — array of `{playerId, category}` at generation time, **no raw welfare**).

EF migration `Sessions`. Indexes: `(TeamId, ScheduledAt)` on `Session`, `(TeamId, GeneratedAt)` on `SessionPlan`. Soft-delete query filter on `Session`.

## B. Generator

Interface `ISessionPlanGenerator` with method
`Task<GeneratedPlan> GenerateAsync(SessionPlanContext ctx, CancellationToken ct)`.

`SessionPlanContext` carries: team id, requested focus (optional), prior session focus + review notes (if any), readiness snapshot (player id + `SafeCategory` only, **no raw**), upcoming session date.

`HeuristicSessionPlanGenerator` (in-process default):
- Pick a focus: explicit override → previous session's stated focus (carry-through) → fallback "General conditioning + skills".
- Choose intensity bucket from the team-wide readiness mix:
  - majority `Ready` / `Monitor` → "Standard"
  - mix includes `ModifyLoad` ≥ 30% → "Reduced"
  - any `RecoveryFocus` ≥ 20% → "Recovery emphasis"
- Build a fixed-shape 5-block plan: warm-up, technical, conditioned game, decision-making, cool-down. Block durations and intent text vary by intensity bucket.
- Append a coach-readable summary referencing the readiness mix in **categories only**.

Pure function; no I/O; deterministic for the same inputs. Registered as `Singleton`.

## C. Controllers

All `[Authorize]`, all use `TeamScope.RequireOwnedTeam` / `RequireOwnedPlayer`.

1. `SessionsController` at `teams/{teamId:guid}/sessions`
   - `GET` list (most recent first, soft-deletes filtered)
   - `POST` create
   - `GET {id}` detail
   - `PUT {id}` update fields (excluding review)
   - `POST {id}/review` set/replace review notes + `ReviewedAt`
   - `DELETE {id}` soft-delete

2. `AttendanceController` at `teams/{teamId:guid}/sessions/{sessionId:guid}/attendance`
   - `GET` list (one row per player on the team; default `Absent` if no record yet)
   - `PUT` bulk upsert: array of `{playerId, status, note?}`. Validates that each `playerId` belongs to the team. Single transaction.

3. `SessionPlansController` at `teams/{teamId:guid}/session-plans`
   - `POST` generate next plan; body optionally `{focus?}` and `{basedOnSessionId?}` else picks the most recent reviewed session
   - `GET` list (most recent first)
   - `GET {id}` detail

## D. Tests

`api/ForgeRise.Api.Tests/Sessions/`:

1. `HeuristicSessionPlanGeneratorTests`
   - All-Ready snapshot → Standard intensity, default focus.
   - Half ModifyLoad → Reduced.
   - Any RecoveryFocus ≥ 20% → Recovery emphasis.
   - Explicit focus override is honoured.
   - Empty roster → still produces a valid 5-block plan with general focus.

2. `SessionEndpointsTests`
   - Create session → list + detail.
   - Review endpoint sets `ReviewedAt` and notes.
   - Soft-delete hides session from list.
   - Non-owner forbidden on every verb.

3. `AttendanceEndpointsTests`
   - Default `GET` returns one row per player as `Absent`.
   - Bulk `PUT` upserts and re-`PUT` updates existing rows in place.
   - Non-team player rejected with 400.

4. `SessionPlanEndpointsTests`
   - Plan generation requires owner.
   - `ReadinessSnapshotJson` contains only ids + categories — no raw fields, no display names not already public.
   - Plan body never contains substrings of any raw welfare fields written via Phase 3 endpoints.
