## Threats considered (STRIDE-ish)

| ID | Threat | Mitigation |
|---|---|---|
| W1 | Coach UI leaks raw welfare | Safe DTOs are the only response from non-`/raw` endpoints. Endpoint tests assert raw substrings (`"tight hamstring"`, `sleepHours`) are absent. |
| W2 | Accidental log of raw via destructuring | `WelfareDestructuringPolicy` redacts any property in `RawWelfareFields.Names` to `[REDACTED]`. New test (`WelfareLoggingTests.Welfare_field_names_are_redacted_in_destructured_payload`) covers it. Hand-rolled welfare logs include only ids + `Category`. |
| W3 | Stranger reads / writes welfare for someone else's team | All endpoints behind `[Authorize]` + `TeamScope.RequireOwnedTeam` / `RequireOwnedPlayer`. Test `Non_owner_cannot_read_or_write_welfare` exercises this. |
| W4 | Raw access untraceable | Every `/raw` GET, every purge, every delete writes a `WelfareAuditLog` row. `GET /teams/{id}/welfare-audit` is owner-only. |
| W5 | "Right to be forgotten" not honoured | `purge-raw` endpoint clears every raw field, sets `RawPurgedAt`, preserves `Category` snapshot. Test `Purge_raw_clears_fields_keeps_category` confirms. |
| W6 | Categorizer overstates readiness on missing data | Default branch returns `Ready` only when concern points = 0. Missing data scores 0 across most axes; test `All_nulls_returns_Ready` documents intentional behaviour. (Tracked as F1 below.) |
| W7 | CSRF on welfare write endpoints | Inherits Phase 2 double-submit middleware; cookie+header verified by `CookieJarHandler` in tests. |
| W8 | Audit table itself tampered with | No update/delete endpoint; controller appends only. EF migration includes the table; ops backups apply. |
| W9 | Storage of "menstrual" fields not yet supported but reserved in `RawWelfareFields.Names` | Schema does not have these columns yet; redaction policy already covers them so a future column won't leak via logs. |

## Findings

| Severity | ID | Finding | Disposition |
|---|---|---|---|
| Low | F1 | When all axes are null, categorizer returns `Ready`. This could under-report risk if a coach sees an all-empty form as "Ready". | Acceptable for iter1 because the readiness endpoint only includes players with at least one check-in. UI must continue to surface "no data" rather than implying readiness. Re-evaluate in Phase 4 when scheduled prompts land. |
| Low | F2 | Audit list capped at 500 most recent entries with no pagination. | Sufficient until we add UI pagination. Tracked. |
| Info | F3 | `WelfareAuditLog` is keyed by `Guid.NewGuid()` (no chain hash). | Out of scope; tamper-evidence is a Phase 5 concern. |
| Info | F4 | `IncidentSeverity` is currently `{Low, Medium, High}` only. | Adequate for coach use; clinical severity scales are out of scope. |
| Info | F5 | NU1902 carry-overs on OpenTelemetry 1.14.0. | Tracked from Phase 1. |

No must-fix items.

## Verdict

✅ Pass — proceed to principal review.
