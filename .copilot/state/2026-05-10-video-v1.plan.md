# Plan — Video Intelligence V1 (skeleton + entities)

> Owner: Planner · Task ID: video-v1 · Date: 2026-05-10

## 1. Problem statement

ForgeRise needs a Video Intelligence module (see
[docs/video-intelligence-architecture.md](../../docs/video-intelligence-architecture.md)).
Before any user-visible video feature can land, we need the module's skeleton
in place: namespaced EF Core entities, a migration, the module's endpoint
shell behind a feature flag, and the storage/queue interfaces that subsequent
slices will plug into. This slice ships nothing a coach can see — it exists
purely to make every later slice additive.

## 2. Acceptance criteria

- [ ] **AC1** — A new `Video` feature folder exists under
  `api/ForgeRise.Api/Features/Video/` with controller(s), DTO folder, services
  folder, storage folder, queue folder.
- [ ] **AC2** — Ten new EF Core entities exist under
  `api/ForgeRise.Api/Data/Entities/Video/` matching §3 of the architecture
  doc (`VideoUploadSession`, `VideoAsset`, `SessionVideoLink`,
  `VideoTimelineEvent`, `VideoClip`, `VideoTag`, `CoachVoiceNote`,
  `TranscriptSegment`, `AiVideoInsight`, `HighlightCandidate`).
- [ ] **AC3** — `AppDbContext` exposes `DbSet<>` for each entity. Every entity
  has a `(TeamId, ...)`-leading index per existing convention.
- [ ] **AC4** — A single EF migration `2026..._VideoModuleSkeleton` exists and
  applies cleanly on a fresh InMemory test DB and on Postgres locally.
- [ ] **AC5** — `IObjectStore` and `IVideoJobQueue` interfaces exist with
  XML-doc method signatures only — no implementations registered in DI yet.
- [ ] **AC6** — A `VideosController` exposes `GET /v1/teams/{teamId}/videos`
  guarded by `TeamScope.RequireOwnedTeam`. When the new
  `Features:Video:Enabled` config key is `false` (default), it returns
  `404 Not Found` so the surface is invisible. When `true`, it returns
  `200 { items: [], total: 0 }`. No other endpoints exist yet.
- [ ] **AC7** — Existing 121 tests still pass. New tests prove AC6 in both
  flag states and prove that a `VideoAsset` row scoped to team A is not
  visible to team B (cross-team isolation smoke test).
- [ ] **AC8** — `dotnet build` clean; `dotnet test` green; CI green.

## 3. Non-goals

- No upload, no streaming, no ffmpeg, no AI, no worker project.
- No web UI; web stays untouched in V1.
- No concrete `IObjectStore` / `IVideoJobQueue` implementations.
- No new third-party packages beyond what ForgeRise.Api already references.
- No welfare-scrub logic yet (no welfare data flows through this slice).
- No RBAC changes — V1 reuses existing team-membership check.

## 4. Approach

**Chosen:** Land the module *inside* the existing `ForgeRise.Api` project
under `Features/Video/` and `Data/Entities/Video/` namespaces, gated by
`Features:Video:Enabled` (default `false`). One EF migration. Two new
interfaces with no impls. One controller with one endpoint. ~10 entity
classes. Unit/integration tests cover the flag and the team-scope.

**Rejected alternatives:**

- _Spin up `ForgeRise.VideoApi` as a separate project now._ Premature — adds
  a build target and CI surface for zero current benefit. The architecture
  already calls for the `VideoWorker` project to be separate (V3); the API
  surface itself can stay in the monolith indefinitely.
- _Skip the feature flag and just ship empty endpoints._ Risk: surfaces a
  "videos" tab in any auto-generated nav; risks inviting users into a
  half-built feature. Flag is one config key, costs nothing.
- _Land entities in one PR, controller in another._ Splits a tiny PR for no
  reviewer benefit and forces the second PR to invent test fixtures the
  first should have shipped.

## 6. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Migration conflicts with an in-flight migration on `main`. | Low | Med | Backend regenerates migration timestamp at PR-rebase time; integration step in CI runs `dotnet ef database update` against an ephemeral Postgres. |
| Feature flag default flips to `true` accidentally via env in dev/prod. | Low | Med | Default in code is `false`; `appsettings.Development.json` may set `true`; production `appsettings.json` and Helm values explicitly set `false`. Integration test asserts default. |
| New entities accidentally exposed via auto-generated OpenAPI / nav scans. | Med | Low | Controller flag-gates with `404`; no entity ships a public DTO yet; no nav entry until V2. |
| Cross-team data leak via missing `TeamId` filter on a future query. | Low | High | Test (d) above seeds team-A data and asserts team-B sees nothing — the test stays in the suite as a regression guard for every later video slice. |
| New entity column types don't translate cleanly to Postgres (`jsonb`, `numeric` precision). | Low | Low | Verify with the local Postgres apply in task 4 before opening the PR. |

## 7. Validation strategy

The Principal Reviewer will verify:

1. **Acceptance criteria** — walks through AC1–AC8 against the diff.
2. **Diff scope** — confirms zero changes to existing endpoints, no new
   third-party packages, no web changes.
3. **Cross-team isolation test** is present and meaningful (not vacuous).
4. **Migration** applies and rolls back on a throwaway Postgres
   (`dotnet ef migrations script 0 2026..._VideoModuleSkeleton` produces a
   sane down).
5. **CI run** on the PR is fully green including security/CodeQL.
6. **Architecture conformance** — entity columns and indexes match
   `docs/video-intelligence-architecture.md` §3.
7. **Security agent sign-off** present in
   `.copilot/state/2026-05-10-video-v1.iter1.security.md`.

## 8. Rollback plan

Single revert. The module is fully gated:

1. `git revert <merge-sha>` reverts code, controller, entities.
2. If the migration has been applied to any environment:
   `dotnet ef database update <previous-migration> --project api/ForgeRise.Api`
   on that environment removes the new tables. Tables are empty by V1
   design, so no data loss.
3. Remove the `Features:Video:Enabled` config key from any env where it was
   set.

No coupling exists with any other slice, so rollback never cascades.
